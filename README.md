# SimpleWebServer（第一版：适合学习）

这是一个 **Linux 下 C++ 轻量级 Web 服务器入门项目**。

这一版我故意做得“简单但完整”，目的是让你先真正看懂下面这些核心模块是怎么串起来的：

- socket
- epoll
- 线程池
- 事件处理
- 日志系统
- CMake 构建

它适合作为你后续继续扩展的 **简历项目第一版**。

---

## 1. 这个项目目前能做什么？

当前版本支持：

1. 监听指定端口，比如 `8080`
2. 接收浏览器连接
3. 解析最简单的 HTTP GET 请求
4. 返回 `www/` 目录下的静态文件
5. 用日志记录连接、请求、错误
6. 用线程池处理客户端请求
7. 用 epoll 做 I/O 多路复用

当前版本 **故意没有做** 的内容：

- keep-alive 长连接
- POST / 登录注册
- 定时器关闭超时连接
- Reactor/Proactor 切换
- ET 边缘触发
- 数据库连接池
- 异步日志
- mmap + writev

这些都很适合你在理解完第一版后继续升级。

## 3. 整体运行流程（先抓主线）

你先记住这个主线：

```text
main
  ↓
创建 WebServer
  ↓
初始化监听 socket
  ↓
创建 epoll，把 listen_fd 放进去
  ↓
进入事件循环 epoll_wait
  ↓
有新连接 -> accept
  ↓
客户端 fd 加入 epoll
  ↓
客户端可读 -> 丢给线程池
  ↓
线程池中的工作线程处理 HTTP 请求
  ↓
读取文件并返回响应
  ↓
记录日志并关闭连接
```

你可以把它理解成：

- **socket** 负责“建立通信”
- **epoll** 负责“盯着谁有事件”
- **线程池** 负责“真正干活”
- **http_connection** 负责“解析请求并构造响应”
- **logger** 负责“把过程记下来”

---

## 4. 各组件是怎么配合的？

## 4.1 main.cpp：程序入口

它做 3 件事：

1. 初始化日志
2. 创建服务器对象
3. 启动事件循环

也就是说，`main` 不处理业务细节，它只负责“把总控拉起来”。

---

## 4.2 server：总调度中心

`WebServer` 是这个项目的中枢。

它主要负责：

- 创建监听 socket
- 设置端口复用
- bind + listen
- 创建 epoll
- 把监听 fd 放进 epoll
- 在事件循环里区分：
  - 是不是新连接
  - 是不是某个客户端可读
- 把客户端处理任务投递到线程池

所以你可以把 `server` 理解成：

> “指挥官，不亲自写业务，但负责调度所有模块。”

---

## 4.3 thread_pool：避免主线程阻塞

为什么要线程池？

因为主线程要一直执行 `epoll_wait`。
如果主线程自己去处理每个请求，那么：

- 读请求
- 解析 HTTP
- 读文件
- 发响应

这些步骤会让主线程忙住，导致它没法及时处理其它连接。

所以这里的做法是：

- 主线程只负责发现“这个 fd 可读了”
- 然后把任务丢进线程池
- 工作线程去处理这个连接

这样主线程可以持续做事件分发。

---

## 4.4 http_connection：处理单个连接

这个类只关心“一个客户端连接怎么处理”。

当前版本做的事情是：

1. 从 socket 读请求
2. 解析请求行，例如：
   ```http
   GET / HTTP/1.1
   ```
3. 判断是不是 GET
4. 映射到 `www/index.html`
5. 读取文件内容
6. 拼出 HTTP 响应报文
7. 发回浏览器

这里你要有一个关键意识：

> 浏览器看到网页，本质上是服务器返回了一段符合 HTTP 协议格式的文本/字节流。

比如响应大概像这样：

```http
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 123
Connection: close

<html>...</html>
```

---

## 4.5 logger：为什么日志是必须的？

因为服务器是长期运行程序。

你不可能每次都靠 `cout` 排查问题。日志的价值在于：

- 知道什么时候启动成功
- 知道谁连接进来了
- 知道请求内容是什么
- 知道哪里报错了

当前版本的日志系统是：

- 单例模式
- 用 `mutex` 保证多线程安全
- 同时输出到终端和文件 `server.log`

后续你可以升级成：

- 异步日志
- 日志分级
- 日志滚动切分

## 5. 构建与运行

## 5.1 环境要求

- Linux（Ubuntu / WSL2 都可以）
- g++ 支持 C++17
- cmake >= 3.10

安装示例：

```bash
sudo apt update
sudo apt install g++ cmake -y
```

---

## 5.2 编译

在项目根目录执行：

```bash
mkdir -p build
cd build
cmake ..
make -j
```

编译成功后会生成：

```bash
./server
```

---

## 5.3 运行

默认运行：

```bash
./server
```

含义：

- 端口：`8080`
- 线程数：`4`
- 静态资源目录：`../www`

也可以手动指定：

```bash
./server 8080 4 ../www
```

格式是：

```bash
./server <port> <thread_num> <root_dir>
```

---

## 5.4 测试

浏览器访问：

```text
http://127.0.0.1:8080/
```

或者命令行测试：

```bash
curl http://127.0.0.1:8080/
```

如果返回 `index.html` 内容，说明服务器已经跑通。

## 6. 第一版里每个技术点分别体现在哪？

### 6.1 socket 在哪里？

在 `server.cpp` 里：

- `socket()`：创建监听 socket
- `bind()`：绑定端口
- `listen()`：开始监听
- `accept()`：接收客户端连接
- `recv()` / `send()`：收发数据

---

### 6.2 epoll 在哪里？

也在 `server.cpp` 里：

- `epoll_create1()`：创建 epoll 实例
- `epoll_ctl()`：把 fd 加进去
- `epoll_wait()`：等待事件发生

你可以把 epoll 理解成：

> “内核帮你盯着很多 fd，谁有事了再通知你。”

---

### 6.3 线程池在哪里？

- `thread_pool.h/.cpp`
- `WebServer::handle_readable()` 中把任务投到线程池

这就是：

```cpp
pool_.enqueue([this, client_fd]() {
    HttpConnection conn(client_fd, root_dir_);
    conn.process();
    del_fd(client_fd);
});
```

意思是：

- 当前有一个客户端 socket 可读
- 不在主线程里处理
- 而是投递给工作线程

---

### 6.4 事件处理在哪里？

事件处理的核心就是 `epoll_wait` 返回后，对不同 fd 和不同事件做不同动作。

在 `run()` 里：

- 如果事件来自 `listen_fd_`，说明有新连接，调用 `handle_accept()`
- 如果某个客户端 fd 可读，调用 `handle_readable()`

这就是最基本的 **事件驱动模型**。

---

### 6.5 日志系统在哪里？

- `logger.h/.cpp`
- 各模块在关键位置调用：
  - 启动成功
  - 新连接到来
  - 请求内容
  - 错误信息

## 8. 你后续可以怎么一步步升级（非常适合学习路线）

建议按这个顺序升级：

### 第一步：看懂当前版本

你至少要彻底理解：

1. 浏览器访问网页时，服务器到底收到了什么
2. 为什么要 `bind/listen/accept`
3. 为什么要 epoll
4. 为什么线程池不是“为了更快”，而是“为了分工和并发处理”
5. HTTP 响应为什么要有状态行、响应头、空行、响应体

---

### 第二步：升级成更像“正式项目”的版本

你可以继续加：

1. **支持长连接 keep-alive**

4. **封装 Buffer 类，避免直接操作裸字符串**
5. **支持 POST**
6. **接入 MySQL，做登录注册**
7. **增加配置文件**
8. **日志异步化**
9. **支持 Reactor / Proactor 模式切换**
10. **使用 mmap + writev 提升静态文件发送性能**

这时候就越来越接近很多经典 C++ WebServer 项目了。

## 11. 这一版的局限性（面试时要诚实）

这只是第一版教学实现，不是生产级服务器。当前问题包括：

- 每个请求处理完就关闭连接
- HTTP 解析很简化
- 没有处理粘包/半包的完整情况
- 没有写缓冲区
- 没有定时器
- 没有连接状态机
- 没有数据库
- 没有高性能文件发送优化

但是作为“学习路径上的第一版”，它非常合适，因为结构已经搭起来了。
